01 快速结论：先建立正确心智模型

当前项目不是“多个微服务”，而是一个以 NestJS 模块组织的模块化单体，加上 Vue 管理后台和独立 EyouCMS 对话模板。系统已经具备智能体、RAG、模型适配、技能、记忆、API 接入和可观测性，但多租户身份与数据库级隔离尚未完成。

模块化单体 一个 NestJS 进程内组合业务模块与调度器
PostgreSQL 权威 业务事实、任务、观测数据与向量 registry
pgvector 双索引 知识片段与图片情景记忆共享基础设施
Redis 非权威 只负责短 TTL 限流与 readiness 探测
已经形成的产品闭环 管理员配置模型 → 建知识库并上传文档 → 创建并发布 Agent → 管理后台、公开页面或 OpenAI 兼容 API 发起对话 → RAG、记忆和技能共同增强回答 → 观测面板统计请求、模型 Token、成本和告警。
不能宣称“完整商用多租户 SaaS” 当前没有 workspace/member/authenticated principal、统一资源租户外键和 PostgreSQL RLS；多数管理 API 没有身份鉴权。公开聊天已使用服务端签名匿名 owner token， 但 bearer token 仍不等于账号或租户身份。

02 总体架构与运行边界

外部请求从 Vue 管理后台、EyouCMS 页面或第三方应用进入同一个 NestJS API。API 内部由应用服务编排数据库、向量、文件、模型和 MCP；后台任务也运行在同一进程。

系统上下文架构图

关键运行特征

同步请求路径

Controller 将 HTTP DTO 转为命令，Application 用例编排领域与端口，Infrastructure 适配器访问 PostgreSQL、Redis、文件和外部服务。

异步任务路径

Knowledge ingestion 和 Agent Memory task 都由进程内 scheduler 轮询 PostgreSQL 任务表；实例横向扩展时依靠数据库领取语义避免重复消费。

部署路径

推荐由 Nginx/HTTPS 提供 Vue 静态文件并反向代理 NestJS；API 连接 PostgreSQL、Redis、本地持久卷和外部模型服务。

架构上的核心取舍 将 pgvector 放在 PostgreSQL 中，牺牲一部分专业向量数据库的独立扩缩能力，换取当前阶段更低的一致性、运维和迁移复杂度； Redis 不承担事实存储，RocketMQ 也不在单体阶段重复实现已有的任务可靠性。

03 DDD 分层、依赖方向与模块组合

后端按业务模块切分，每个模块内部使用 Domain、Application、Infrastructure、Presentation 四层。抽象端口位于 Application，适配器位于 Infrastructure，并在 NestJS Module 中完成绑定。

Presentation Controller、DTO、Validation、HTTP/SSE/OpenAI 协议转换；不得承载业务规则。
↓ 调用用例 / 应用服务
Application Use Case、编排服务、Repository / Gateway / Storage / Index 抽象端口；只依赖 Domain 与抽象。
↓ 使用领域模型；Infrastructure 反向实现端口
Domain 实体、值类型、状态、纯函数和业务不变量；不依赖 NestJS、TypeORM、HTTP 或外部 SDK。

  • 维护 draft / published / disabled 状态。
  • 校验模型供应商、知识模块和技能是否存在。
  • 通过多对多关系复用 Knowledge Module 和 Skill。
  • 公开目录只返回 id、name、description。

关键代码

  • domain/agent.ts：状态与领域数据结构。
  • application/agent-catalog.service.ts：跨模块查询入口。
  • application/*agent*.use-case.ts：命令用例。
  • infrastructure/typeorm-agent.repository.ts：持久化。

公开发布边界 正式 API、公开目录和公开聊天都只允许 published；draft 不进入公开目录且公开聊天返回 422，disabled 保持停用。后台测试入口仍可验证草稿 Agent。
Chat：整个系统的运行时编排中心

ChatWithAgentUseCase 的执行顺序

  1. 将 agentId、conversationId、source 写入观测上下文。
  2. 读取 Agent，并按 admin/api/public 校验可用状态。
  3. 并行读取模型配置、知识检索、技能集合、短期/长期记忆和图片情景记忆。
  4. 拼装 system prompt、知识片段、长期记忆和“不可信图片证据”约束。
  5. 读取当前附件；图片转 image_url，音频转 input_audio。
  6. 有 MCP 工具时进入工具环；否则调用流式模型。
  7. 多模态模型在首个输出前失败时，降级为纯文本请求。
  8. 流成功结束后更新对话数，并尽力记录普通记忆和图片情景任务。
故障隔离 普通记忆与图片情景记忆写入被独立 try/catch 包裹，失败只记录告警，不回滚已经成功返回给用户的聊天。
三个协议入口 POST /api/agents/:id/chat、POST /api/public/agents/:agentId/chat、POST /api/v1/chat/completions 最终复用同一用例。
Knowledge：上传、处理与检索

写入链路

浏览器创建 Upload Session，逐片流式写入本地存储；Complete 校验分片数量、总字节数和可选 SHA-256，合并文件，并在数据库事务中创建 Document 与 Ingestion Job。

查询链路

按 Agent 绑定的 moduleId 分组到 Knowledge Base，使用每个 Base 的 embedding 模型生成查询向量，每库取 4 条，跨库按 score 排序后最多返回 8 条。

  • Document 状态：queued → processing → ready | failed。
  • 解析器通过 DocumentTextExtractor 端口组合具体格式实现。
  • TextChunker 负责将文本切分为稳定的 chunk。
  • claim 使用 FOR UPDATE SKIP LOCKED；lockOwner 限制完成/失败/进度更新，超时 lease 由 scheduler reclaim。
  • attempts/maxAttempts、nextRunAt 和封顶指数退避控制失败重试。
  • 批量 embedding 与向量 upsert 大小由配置控制；chunk ID 由 documentId + index 稳定哈希，重试不会重复插入。
  • 检索只接受 ready 文档，processing/failed 向量被 SQL 排除。
  • 删除文档/模块/知识库时同步清理文件、任务、关系和向量。
Agent Memory：三类记忆与可靠图片情景任务

短期线程

按 conversationId + ownerKey 存储最近消息，恢复同一会话上下文并进行重叠去重。

稳定事实 / 偏好

从“记住”、姓名、喜欢、不喜欢、希望等明确文本规则提取 semantic/preference 记忆。

图片情景

保存图片 artifact 引用，异步调用多模态模型提取结构化描述，再 embedding 并写 pgvector。

可靠性机制

  • 幂等键由 agentId、ownerKey、conversationId、排序后的 attachmentId 和用户文本哈希构成。
  • 每个 memory + kind 存在唯一任务；任务分 extract 与 index。
  • PostgreSQL 使用 FOR UPDATE SKIP LOCKED 并发领取。
  • 任务有 lockOwner、lockedAt、attempts、maxAttempts、nextRunAt。
  • 失败使用封顶指数退避；达到最大次数进入 dead。
  • embedding 先保存到任务，再写向量，减少索引失败后的重复模型调用。
  • 启动和周期 tick 回收超时 lease，并巡检 memory、task、artifact、媒体和索引一致性。
  • 所有管理、读取、重试、修复都要求 agentId + ownerKey。
匿名 owner 可信派生 Admin/Public 浏览器只持有服务端签发的 HMAC bearer token；Controller 校验签名后以 SHA-256 派生内部 ownerKey，客户端不能直接构造或篡改 ownerKey。token 被窃取后仍可重放，且不代表 workspace/member 身份。
Model Providers：密钥、兼容网关与成本观测
  • 配置时先请求供应商 /models 验证连通性。
  • 密钥使用 AES-256-GCM 保存，根密钥来自 CREDENTIAL_ENCRYPTION_KEY。
  • Runtime Service 只在服务端解密，API 返回摘要而不返回原始密钥。
  • Gateway 支持流式 chat、非流式 chat、embedding 和 function calling。
  • 请求由 AbortSignal.timeout 限制超时。
  • 优先读取模型 usage；没有 usage 时按文本估算 token。
  • 成本按“每百万 Token 单价”换算为 costUsdMicros。
Skills：Prompt 指令与 MCP 工具
  • prompt 技能的 content 拼入 system prompt。
  • mcp 技能保存 endpoint、私密 headers 和工具定义缓存。
  • 安装/刷新时执行 MCP initialize 和 tools/list。
  • 对话中模型返回 tool_calls 后，由 MCP tools/call 执行并回填结果。
  • 同名工具以先绑定技能为准；停用技能不参与运行时。
  • 工具参数只接受 JSON object；解析失败时使用空对象。
  • 最大工具轮数来自 SKILL_TOOL_MAX_ROUNDS，默认设计值为 5。
  • 当前只支持 Streamable HTTP，不支持 stdio MCP Server。
API Access、Observability、Branding 与 Health

API Access

  • 只有 published Agent 可创建应用。
  • 原始 ag_live_... 密钥只返回一次。
  • 数据库只保存 SHA-256 hash 与 maskedKey。
  • 正式 API 使用已认证 application.id 作为内部 ownerKey。
  • Redis key 中 identifier 先做 SHA-256。

Observability

  • HTTP 请求注入 traceId/spanId，并返回 X-Trace-Id。
  • 记录 HTTP、model、tool 三类事件。
  • 展示请求量、错误率、平均/P95 延迟、Token、成本和进程内存。
  • 慢请求、慢模型、高成本和错误产生告警。

Branding

  • 全局单例 brand settings。
  • 软件名称保存在 PostgreSQL。
  • 图标由本地存储适配器管理。
  • Vue 和 EyouCMS 页面读取同一公开品牌接口。

Health

  • /api/health 只做 liveness，不访问依赖。
  • readiness 检查 PostgreSQL、vector extension 和 Redis。
  • 未配置 Redis 返回 disabled，仍可 ready。
  • 配置但不可用则返回 not_ready / 503。

Redis 与文件存储边界

Redis

  • Lua：INCR + 首次 PEXPIRE。
  • 关闭 offline queue 与自动重连。
  • key 不含原始 API key、ownerKey 或 IP，而是 SHA-256。
  • 配置 Redis 后故障时限流 fail-closed。
  • 未配置时仅使用进程内窗口，生产配置校验要求 Redis。

本地文件

  • Knowledge storage 流式写入、合并并计算 SHA-256。
  • Chat attachment 校验 UUID、MIME、文件签名、大小和路径。
  • 附件二进制与 JSON metadata 分开保存。
  • 路径通过 root prefix 校验防止目录穿越。
  • 生产横向扩容前应迁移到共享对象存储。

10 安全边界与商用 SaaS 差距

当前系统已经实现若干重要的单体安全措施，但“资源隔离键”与“经过认证的租户身份”不是同一件事。 面向外部客户上线前，必须先完成 workspace、principal、授权和 RLS。

已实现的防护

密钥 模型密钥使用 AES-256-GCM；API key 只保存 SHA-256；Redis key 不写原始 identifier；浏览器不持有模型密钥。
输入与文件 DTO 白名单校验；附件限制 MIME、签名、大小和 UUID；文件 storage key 防止目录穿越。
记忆 agentId + ownerKey 贯穿数据库、artifact、原图和向量；浏览器 token 由 HMAC 校验后派生 ownerKey，图片被明确标记为不可信外部证据。
可靠性 PostgreSQL 权威任务、幂等键、唯一约束、lease、退避、dead、恢复和巡检减少重复与不一致。
日志 结构化日志不写完整 prompt、回复、附件或 base64；任务错误专门脱敏 bearer、key 和路径。
高成本入口 Public Chat 和 API application 有限流；Redis 配置后故障时 fail-closed。

必须正视的高优先级风险

风险 当前事实 可能后果 建议优先级
管理 API 无认证 Agent、模型、知识、Skill、API key、观测和品牌写接口没有统一 Guard 未受信网络可读取、修改配置或签发密钥 P0
没有 workspace 根模型 核心资源表没有统一 workspaceId 无法形成客户级数据和计费边界 P0
没有 PostgreSQL RLS 隔离主要依赖应用查询条件 遗漏条件即可跨租户读取或修改 P0
匿名 bearer token 可重放 已阻止 ownerKey 伪造，但 token 仍保存在浏览器且不是账号身份 token 被窃取后可访问同 owner scope，无法表达成员权限 P1
本地文件不适合横向扩容 文档、附件、图标依赖单机路径或持久卷 多副本间文件不可见、备份和迁移困难 P1
全局观测与成本 无 workspace 成本归属、预算与配额 无法计费、限制滥用或提供客户账单 P1
单根加密密钥 模型凭证依赖单个环境变量密钥 缺少 KMS、版本、轮换和逐租户 envelope encryption P1
缺少审计与恢复演练 没有不可抵赖管理审计、备份恢复验证 难以满足企业治理和事故追溯要求 P1
上线结论 当前版本适合受控内网、单组织部署或继续产品验证；在完成管理认证、workspace 绑定、可信 principal 和 PostgreSQL RLS 前，不应作为不受信任客户共享的多租户 SaaS 对外承诺安全隔离。

11 建议演进路线与中间件触发条件

演进应优先补“安全与商业闭环”，而不是堆叠更多基础设施。RocketMQ、专业向量数据库和微服务拆分都应由真实负载与组织边界触发。

P0 · 多租户安全

  1. Workspace / Member / Role 模型。
  2. 统一 authenticated principal。
  3. 管理、API、Public 三类身份边界。
  4. 所有核心资源增加 workspaceId。
  5. API key → application → workspace 映射。
  6. 可信 public workspace/user resolution。
  7. PostgreSQL RLS 与 transaction-local context。
  8. 默认 workspace 回填 migration。
  9. 非超级用户 RLS E2E。

P1 · 商业运营

  1. 对象存储替换本地文件。
  2. workspace Token、成本、配额和预算。
  3. 审计日志与密钥轮换。
  4. Retention / forgetting / recycle bin。
  5. 用户记忆纠错与管理 UI。
  6. 备份、恢复和灾备演练。
  7. SQLite/Zvec 历史生产迁移工具。

P2 · 规模化

  1. 模型 Worker / 索引 Worker 独立部署。
  2. PostgreSQL Outbox publisher。
  3. RocketMQ 至少一次投递 + 幂等消费。
  4. pgvector 反向孤儿扫描。
  5. 读写分离和多区域。
  6. 专业向量库迁移评估。
  7. 多模态 embedding 与召回评测。

何时引入 RocketMQ

判断条件 当前状态 结论
任务是否已经跨独立服务 否，scheduler 和 API 同进程 继续 PostgreSQL task table
是否需要多个独立订阅者 暂无 无需消息总线
是否需要跨服务削峰 当前负载未证明 先观测数据库任务吞吐
是否需要独立延迟/死信治理 数据库已有 nextRunAt / dead 现有能力足够
模型/索引 Worker 是否独立扩容 未来可能 达到此条件再引入
未来消息语义 即使接入 RocketMQ，也应由 PostgreSQL Outbox 发布版本化事件，携带 workspaceId、traceId 和 message key；采用至少一次投递与幂等消费者， 不能将 RocketMQ 事务消息表述为 exactly-once。

何时迁移专业向量数据库

  • 向量规模或 QPS 明显超过 PostgreSQL 业务负载可接受范围。
  • 需要跨区域向量复制、独立资源隔离或更复杂的混合检索。
  • 向量团队需要独立发布、扩缩和 SLO。
  • 经过基准测试证明迁移收益大于双写、回填、校验和运维成本。

现有 VectorIndex 与 AgentMemoryIndex 端口已为迁移保留边界；真正困难的是生产数据迁移、一致性校验和回滚，而不是替换类名。
